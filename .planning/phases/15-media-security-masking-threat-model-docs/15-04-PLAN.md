<threat_model_substance>
<!-- Write this into docs/SECURITY.md as a structured asset/threat/mitigation subsection. Source: 15-RESEARCH
     "Parent-Realm Object-URL Threat Model" (rows verified against the Phase-14 design). -->
Asset: the blob: object URL minted in the PARENT (renderer-owning) realm by URL.createObjectURL(mediaSource) for
  adaptive playback (hls.js/MSE), assigned to the inert in-iframe <video>.src; the parent fetches all segments.
Rows (asset / threat / STRIDE / why mitigated):
  1. Mirrored content scripts the page to read+exfiltrate the blob -> Info-disclosure -> sandbox is exactly
     allow-same-origin, NEVER allow-scripts; a blob: is only readable via fetch/XHR/FileReader (all need script);
     no script -> child can play but not read. createViewer reads back the token (viewer-sandbox-invalid).
  2. Parent-origin object URL reachable by the child for non-media use -> Info-disclosure/Elevation -> the blob's
     origin is the parent document's, not the mirrored page's; no script to dereference it; CSP media-src blob:
     permits only media LOADING, not fetch/connect (there is NO connect-src).
  3. Leaked/long-lived object URL persists (UAF/leak/cross-session bleed) -> Info-disclosure/DoS -> revoked on
     destroy/destroyAll; destroyAll() runs before any new-identity snapshot document swap; a revoked blob: is dead.
  4. blob: widens CSP enough to load arbitrary local resources -> Tampering -> blob: is scoped to media-src ONLY
     (not img-src, not default-src); default-src 'none' + no script-src/connect-src means blob: cannot script/XHR.
  5. The parent realm itself is the privileged target -> Elevation -> the parent is renderer/host code, never
     sandboxed, trusted by construction; the threat boundary is mirrored-content -> host, which IS the sandbox;
     moving the player to the parent is precisely what AVOIDS granting the sandbox allow-scripts.
Plain-language worst case (state verbatim-style): "The child iframe plays the parent's media object URL but cannot
  script, read, copy, or exfiltrate it — there is no allow-scripts, no connect-src, and the blob: is dead the moment
  the player is destroyed. An attacker who fully controls the mirrored page gains the ability to display media the
  host already chose to mirror, and nothing more."
Backing tests (cite): sandbox-token unchanged (createViewer viewer-sandbox-invalid + purity scan); allow-scripts
  absent from media-player.js (Plan 03 named case + purity glob); blob: scoped to media-src, no script-src/connect-src
  (renderer-media-csp.test.js); object-URL revoke on destroy/destroyAll (renderer-media-player.test.js / Plan 03 cite).
