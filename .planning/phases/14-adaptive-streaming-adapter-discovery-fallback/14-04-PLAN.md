<objective>
Make the adapters surface manifest URLs that never appear as a plain element `src` (MADPT-02). Add **opt-in, off-by-default** network observation: the Playwright adapter via `page.on('response')` (and/or the CDP `Network` session) and the extension adapter via `chrome.webRequest.onCompleted` (and/or `chrome.debugger` Network), filtering `.m3u8`/`.mpd` by the pure `classifyManifest` helper (extension OR content-type), correlating best-effort, and emitting `STREAM.MEDIA_HINT` through the existing `transport.send` path. Absence of an opt-in degrades gracefully — neither adapter registers a listener, and the progressive path is untouched.

Purpose: This is the discovery slice. Because both adapters build their relay allow-set from `Object.keys(STREAM)`, adding the op in Plan 01 already makes it relayable — this plan only adds the EMISSION code. Correlation is best-effort by contract (single-active → element scope; ambiguous → page scope; the viewer matches a page hint to an MSE-opaque element on play, Plan 03) — never block on perfect correlation.

Output: opt-in manifest observation in both adapters + extended mock-page / fake-chrome tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| captured-page network → adapter | Manifest URLs observed on the driven page (attacker-influenced content) are turned into hint payloads |
| adapter → wire | The hint rides the existing raw relay (1 MiB cap, envelope unchanged); the op is auto-allowlisted from Object.keys(STREAM) |
| extension permission surface | `chrome.webRequest`/`chrome.debugger` are powerful — must be required ONLY when discovery is opted in |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-14 | Info-disclosure / Elevation | Always-on network observation widening the adapter's surface/permissions | mitigate | Discovery is OPT-IN, off by default (locked); no listener registered and no permission required unless explicitly enabled (both tasks assert no-opt-in → no listener); `validateChrome` requires `chrome.webRequest` ONLY when opted in. |
| T-14-15 | Tampering | A hostile/oversized manifest URL emitted as a hint | mitigate | The emitted `manifestUrl` is re-gated at the VIEWER (`gateAssetUrl`, Plan 03) before any fetch; the hint rides the 1 MiB-capped raw relay; classifyManifest is try/catch-guarded (Plan 01) so a malformed url cannot wedge the observer. |
| T-14-16 | Spoofing / Tampering | A page-scope hint mis-attributed to the wrong element | mitigate (best-effort by contract) | Correlation defaults to page scope when ambiguous (never asserts a wrong element); the viewer matches a page hint only to an MSE-opaque element on play (Plan 03); identity stamps + isCurrentStream bound it to the current stream. |
| T-14-SC | Tampering | npm/pip/cargo installs in this plan | mitigate | This plan installs NOTHING (adapter emission + tests only; mock `page`/fake `chrome`). No new dependency; `dependencies` stays `{ ws }`. No `[ASSUMED]`/`[SUS]` install. |
</threat_model>
