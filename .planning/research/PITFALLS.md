# Pitfalls Research

**Domain:** DOM-mirroring framework extraction (library extraction from production app + defensive HTML rendering + CSSOM capture + npm SDK publication + systems-paper evaluation)
**Researched:** 2026-06-09
**Confidence:** HIGH for security/CSSOM/MV3/npm items (verified against official docs, rrweb docs/issues, vendor write-ups); MEDIUM for extraction-process and evaluation items (multiple credible sources, partly practice-based)

Defensive research for PhantomStream's maintainers: the mistakes below are what projects
in this exact niche (rrweb, session-replay vendors, extracted-from-product SDKs) have
documented getting wrong. File references point at this repo's own code where the pitfall
is already latent.

---

## Critical Pitfalls

### Pitfall 1: Losing production hardening during the "clean" rewrite

**What goes wrong:**
The extracted module is rewritten "cleanly" and silently drops battle-tested behavior that
existed for non-obvious reasons: the single-pass layout read before truncation (collapses
N forced reflows into 1), the `setTimeout`-chain watchdog (not `setInterval`, so cadence
resets on drain), whole-subtree-only truncation, rAF-matched flushing, the
`_pendingStreamStart` parking logic, default-value elision in style capture. Each looks
like incidental complexity to a fresh reader; each encodes a production incident
(`docs/DESIGN-HISTORY.md` records the 45 s YouTube serialize that forced curated styles).

**Why it happens:**
Extraction invites refactoring; refactoring invites simplification; the reasons live in
docs and commit history, not in the code's shape. The reliability inventory
(`docs/ARCHITECTURE.md` §5) lists nine failure-mode defenses — none enforced by any
runnable test in this repo today (CONCERNS.md: capture core has zero test coverage here).

**How to avoid:**
- Treat `docs/ARCHITECTURE.md` §5 as an explicit acceptance checklist: every extracted
  module PR states which defenses it carries and where.
- Extract-then-refactor, never rewrite-from-spec: the first commit of each module is a
  near-verbatim lift with only transport/options seams changed, verified against the
  reference; refactors come afterwards, with the oracle (Pitfall 2) in place.
- Encode performance lessons as tests: snapshot-time budget on a heavy fixture page; a
  forced-reflow-count assertion (spy on `getBoundingClientRect`) during truncation.

**Warning signs:**
- PR descriptions containing "simplified the watchdog/truncation/flush logic."
- Extracted serializer loses `STYLE_DEFAULTS` elision or the rect pre-pass Map.
- `staleFlushCount` / `truncated` / `missingDescendants` telemetry disappears from the wire.

**Phase to address:**
Capture extraction phase (first lift), with the reliability checklist as exit criteria.

### Pitfall 2: No regression oracle — behavior drift vs. the reference implementation

**What goes wrong:**
The extracted serializer produces output "mostly the same" as the reference — different
attribute ordering, nid assignment order, style elision — and nobody notices until the
FSB swap-in fails or the paper's ablation compares new code against numbers generated by
old code. Drift compounds: by detection time you can't separate intentional fixes from bugs.

**Why it happens:**
The reference exists (`reference/`, pinned to FSB `867d6f0c`) but is not executable as an
oracle: its tests depend on FSB's harness (`fsb-test-utils`, jest mocks) absent from this
repo. Without automated A/B comparison, "matches the reference" is a vibe.

**How to avoid:**
- Build a **differential harness early** (before extracting the serializer): load a frozen
  fixture page headlessly, run reference `dom-stream.js` and the extracted capture core
  side by side, structurally diff snapshots and diff-op streams (normalizing legitimately
  changed fields such as sanitization stripping).
- Keep an **intentional-divergence ledger**: every deliberate difference from the
  reference (on* stripping, WeakMap identity, no `data-fsb-nid` stamping) gets a tracked
  entry; differential-test exceptions must cite a ledger entry.
- This is classic characterization / golden-master testing for legacy extraction: pin the
  reference's observable behavior before changing anything.

**Warning signs:**
- Extracted-module tests assert hand-written expected output instead of
  reference-generated output.
- The fixture corpus is "whatever page was handy" rather than frozen and versioned.
- FSB swap-in scheduled before any differential run has happened.

**Phase to address:**
Capture extraction phase (harness first); re-run at renderer extraction and as the FSB
swap-in gate.

### Pitfall 3: Sanitize-once thinking — mXSS through the serialize→parse roundtrip

**What goes wrong:**
The pipeline is intrinsically serialize→transmit→parse: capture serializes live DOM to
HTML strings; the viewer re-parses with `innerHTML`
(`reference/dashboard/dashboard.js:3242` does exactly `temp.innerHTML = m.html`).
Sanitizing only at capture is defeated by **mutation XSS**: markup inert in the
capture-side parse mutates into executable form when re-parsed in the viewer. DOMPurify's
own docs flag `div.innerHTML = DOMPurify.sanitize(html)` as mXSS-prone by design (the
extra serialize-parse roundtrip); PortSwigger has published repeated DOMPurify bypasses
built on exactly this. PhantomStream additionally cannot trust the capture side at all —
the malicious *page* is in the threat model, and capture code runs inside that page's
context where it can be tampered with.

**Why it happens:**
"We strip `on*` and `javascript:` at capture, so render is safe" feels complete. It isn't:
(a) mXSS payloads survive capture-side stripping via parser differentials; (b) capture
runs in attacker-controlled context; (c) any future serialization path (on-demand subtree
fetch, a new side channel) silently bypasses sanitization unless it is a pipeline *stage*
with a single chokepoint, not a function sprinkled at call sites.

**How to avoid:**
- **Both ends, plus sandbox as the backstop.** rrweb's documented position is correct:
  filtering "is never a complete solution"; the replay iframe's `sandbox` (no
  `allow-scripts`) is the primary defense, with `<script>` additionally rewritten to
  `noscript`. Adopt the same layering: capture-side strip (`on*`, `javascript:`/`data:`
  URLs in URL attributes, `<script>`, iframe `srcdoc` attributes, `<meta http-equiv>`),
  render-side sanitize *again* before insertion, sandboxed iframe regardless.
- Render-side: sanitize into a DOM (DOMPurify `RETURN_DOM_FRAGMENT`) and insert nodes —
  never sanitize-to-string-then-innerHTML, which reintroduces the roundtrip.
- One named chokepoint per side so the three current serialization paths (snapshot,
  `processAddedNode`, attr re-absolutification) and future ones cannot skip it.
- The native Sanitizer API is not a dependable baseline (Chrome 146 / Firefox 148,
  limited availability as of early 2026): feature-detect at most; DOMPurify is the
  render-side library.
- Avoid DOMPurify config foot-guns documented by its maintainers and researchers:
  loosened `ALLOWED_URI_REGEXP`, `ADD_URI_SAFE_ATTR`, `CUSTOM_ELEMENT_HANDLING`, and
  hooks using `forceKeepAttr` each create full-bypass risk.

**Warning signs:**
- Any renderer path reaching `innerHTML`/`insertAdjacentHTML` without the sanitizer
  chokepoint.
- No mXSS fixtures in the test suite (namespace-confusion payloads, `<svg>`/`<math>`
  mutation vectors).

**Phase to address:**
Sanitization pipeline phase — must land **before** npm publication; render-side
sanitization belongs to renderer extraction.

### Pitfall 4: Iframe sandbox misconfiguration — the `allow-scripts allow-same-origin` trap

**What goes wrong:**
The viewer iframe either has no `sandbox` at all (current state — CONCERNS.md confirms
`previewIframe` sets none) or later gains `allow-scripts` for a feature while keeping
`allow-same-origin`. That pair lets framed content escape the sandbox entirely: script +
same-origin means the framed document can reach `window.parent`, find its own iframe
element, and `removeAttribute('sandbox')`. The HTML spec and MDN warn the combination
"effectively enables an embedded page to break out of all sandboxing." With `srcdoc`, the
inherited origin is the *viewer's* origin — escape means full XSS in the consumer's app.

**Why it happens:**
`allow-same-origin` is required so the parent renderer can script the mirror document
(apply diffs, position overlays). Then a feature "just needs" `allow-scripts` — rrweb's
`UNSAFE_replayCanvas` does exactly this and is documented as unsafe. The renderer is
becoming an embeddable component; consumers may also pass their own iframe, which the
framework cannot assume is sandboxed.

**How to avoid:**
- Renderer **creates and owns** the iframe: `sandbox="allow-same-origin"` (plus at most
  `allow-forms`), never `allow-scripts`. Enforce in code: a startup assertion that the
  sandbox lacks `allow-scripts`, refusing to render otherwise.
- Document the invariant as non-negotiable in public API docs; any future
  script-execution feature must be a separately named `UNSAFE_` opt-in (rrweb's naming
  makes the risk legible).
- Belt-and-suspenders: inject a restrictive CSP `<meta>` into the constructed `srcdoc`
  document (`script-src 'none'`) so a sandbox regression still doesn't execute scripts.
- Keep `pointer-events:none` neutering on nested iframes in the mirror (reference does
  this) — nested live iframes are otherwise an unsandboxed hole.

**Warning signs:**
- Any diff adding `allow-scripts` to the sandbox list.
- Consumer-supplied iframe accepted without sandbox validation.
- A `<canvas>` live-replay feature request (the classic pressure point).

**Phase to address:**
Renderer extraction phase (sandbox enforced at creation); sanitization pipeline phase
(CSP meta injection).

### Pitfall 5: CSS treated as styling, not as an attack and exfiltration channel

**What goes wrong:**
Snapshot inline `<style>` blocks — and, in the planned CSSOM mode, raw rule text — are
injected verbatim into the viewer document (`reference/dashboard/dashboard.js:2789–2791`,
no filtering). Malicious CSS can exfiltrate viewer-context data via attribute-selector +
`url()` beacons, load attacker fonts/images (tracking, request forgery from the
consumer's origin), and pull attacker stylesheets at render time via `@import`. CSSOM
capture mode *increases* this surface — it ships more rule text by design.

**Why it happens:**
Sanitization effort concentrates on HTML/JS; CSS "can't execute code" so it gets a pass.
But the mirror renders in the consumer's origin context where CSS-driven requests carry
ambient authority.

**How to avoid:**
- Add a CSS pass to the sanitization stage: strip/rewrite `@import`, filter `url()`
  values to http(s) with an optional origin policy, drop `expression(` and
  `url(javascript:` legacy vectors (already recommended in CONCERNS.md).
- The render-side CSP meta in the srcdoc document is the systemic control: scope
  `style-src`/`font-src`/`img-src` deliberately. What the mirror may load (images
  probably yes) is a documented fidelity-vs-privacy tradeoff — paper discussion material.
- In CSSOM mode, flatten `@import` chains at capture (required anyway — Pitfall 6), which
  removes render-time import fetches entirely.

**Warning signs:**
- `<style>` text or rule text concatenated into srcdoc without passing the sanitizer.
- No CSP meta in the constructed mirror document.

**Phase to address:**
Sanitization pipeline phase (CSS pass); CSSOM capture phase (import flattening).

### Pitfall 6: CSSOM capture assumes `cssRules` is readable and `<style>` text is truthful

**What goes wrong:**
The stylesheet-centric capture mode hits three browser realities that every session-replay
vendor has tripped over:

1. **Cross-origin sheets throw.** Reading `.cssRules` on a stylesheet loaded from another
   origin without CORS throws `SecurityError` — by spec, not a bug. rrweb-snapshot's
   issue tracker documents this exact failure (`styleSheets.cssRules have CORS problem`,
   rrweb-snapshot #45). A naive `for (const sheet of document.styleSheets)` loop dies or
   skips most real-world sheets (CDNs, font providers).
2. **Dynamically inserted rules are invisible to text serialization.** CSS-in-JS libraries
   in production mode (styled-components/emotion "speedy" mode) inject rules via
   `insertRule()` into `<style>` elements whose **text content stays empty**. Serializing
   `<style>` textContent or innerHTML captures nothing; the CSSOM is the only source of
   truth. Symptom: mirror looks fine on dev builds, unstyled on production builds.
3. **`adoptedStyleSheets` / constructable sheets exist outside the DOM entirely.** They
   never appear in `document.styleSheets` serialization or innerHTML, are heavily used by
   web components, and constructable sheets **reject `@import`** (`replace()`/
   `replaceSync()` drop them with a console warning; `insertRule` throws) — so replaying
   captured rules into constructable sheets requires `@import` chains to be flattened at
   capture.

**Why it happens:**
The mental model "stylesheets are text files the page links" is wrong for the modern web.
The CSSOM is a live mutable object graph, partially access-restricted, partially detached
from the document tree.

**How to avoid:**
- Per-sheet **fallback chain**: try `cssRules` → on `SecurityError`, fall back to
  re-linking the `href` (current reference behavior) → optionally fetch the CSS with CORS
  where the host adapter permits (extension contexts can use host permissions; CDP can
  use `CSS.getStyleSheetText`) → last resort, computed-style capture for affected
  subtrees. Wrap every `cssRules` access in try/catch; never let one sheet kill capture.
- Serialize from the **CSSOM, not element text**: iterate rules per sheet; include
  `document.adoptedStyleSheets` and (with shadow DOM mirroring) each shadow root's
  `adoptedStyleSheets`.
- Flatten `@import` at capture into a linear rule list (preserving media conditions as
  `@media` wrappers).
- Test fixtures must include: a production-built styled-components page, a web-components
  page using constructable sheets, and a page with cross-origin CDN CSS.

**Warning signs:**
- Capture code reads `sheet.cssRules` without try/catch.
- The CSSOM-mode fixture corpus contains only dev-built or same-origin pages.
- Mirror styling diverges between `npm run dev` and production builds of the same fixture.

**Phase to address:**
CSSOM capture mode phase; the ablation in the evaluation phase should measure the
fallback-chain hit rates per corpus page (good paper data).

### Pitfall 7: CSSOM mode without CSSOM *mutation* tracking recreates the drift it was built to fix

**What goes wrong:**
MutationObserver does **not** observe CSSOM mutations: `insertRule`/`deleteRule`,
`CSSStyleSheet.replace()`, or assignment to `adoptedStyleSheets` produce no mutation
records. A CSSOM capture mode that snapshots stylesheets once and then relies on the
existing diff stream will freeze styles at snapshot time — exactly the frozen-style drift
(limitation #1) the mode exists to fix, just moved one level up.

**Why it happens:**
The diff pipeline's mental model is "MutationObserver sees everything." It sees the DOM,
not the CSSOM. rrweb handles this by monkey-patching `CSSStyleSheet.prototype.insertRule`
/ `deleteRule` / `replace` / `replaceSync` and the `adoptedStyleSheets` setters, emitting
dedicated style-mutation events.

**How to avoid:**
- Budget for a **style-ops channel** in the protocol (e.g., `style-insert`,
  `style-delete`, `style-replace` addressed by a stable sheet ID) and patch the CSSOM
  mutation methods in the capture core, same pattern as the existing dialog
  monkey-patches.
- Stable sheet identity is a sub-problem: sheets have no DOM node to stamp; key them in a
  WeakMap with minted sheet IDs (aligns with the WeakMap node-identity rework).
- If v1 scopes this out, document loudly that CSSOM mode handles *static* sheets plus
  class-flip diffs, and that pages mutating CSSOM at runtime still drift — and say so in
  the paper rather than letting a reviewer find it.

**Warning signs:**
- Protocol spec for CSSOM mode has snapshot messages but no incremental style op.
- A theme-toggle fixture (class flip) passes but a `insertRule`-driven animation fixture
  drifts.

**Phase to address:**
CSSOM capture mode phase (protocol + patching); explicitly scoped in the paper's
limitations if deferred.

### Pitfall 8: WeakMap identity rework breaks wire addressing in subtle halfway states

**What goes wrong:**
Replacing `data-fsb-nid` stamping with WeakMap identity changes the *addressing contract*,
not just storage. Failure modes: (a) the serializer stops stamping but the renderer still
resolves via `querySelector('[data-fsb-nid]')` — every diff op misses; (b) ID mappings
desync between capture and render after re-snapshots, truncation recoveries, or node
moves (a moved node must keep its ID; a naive "assign on serialize" re-mints); (c) the
recorder and renderer disagree about who owns ID minting. rrweb hit exactly this class of
bug: storing serialized IDs on nodes (`node.__sn`) caused ID collisions between recorder
and replayer sharing a context, fixed by making a single `Mirror` abstraction the source
of truth (rrweb PR #868).

**Why it happens:**
The data-attribute scheme conflated three things that now separate: identity storage
(attribute), wire addressing (attribute value), and renderer lookup (selector query).
WeakMap replaces only the first; the other two need explicit replacements.

**How to avoid:**
- Mirror rrweb's proven shape: a `Mirror` object on each side — capture: `WeakMap<Node,
  nid>` + `Map<nid, Node>`; render: `Map<nid, Node>` maintained incrementally on
  `add`/`rm`. This also kills the O(M×N) `querySelector`-per-op renderer hot path
  (CONCERNS.md performance item) in the same stroke.
- Define ID lifecycle rules in the protocol doc: when IDs are minted, that moves preserve
  them (childList removal+addition of the same node must be detected as a move or
  handled idempotently), what resets them (new snapshot/session).
- Bonus to bank: WeakMap can key text nodes directly, removing the parent-nid indirection
  for `text` ops — decide deliberately whether to take the protocol change now or keep
  parent-addressing for FSB wire compatibility.
- Differential harness (Pitfall 2) catches halfway states: reference and extracted runs
  must produce structurally equivalent op streams against the fixture corpus.

**Warning signs:**
- Renderer source still contains `[data-fsb-nid` selector strings after the rework.
- `staleMutationCount` spikes on fixtures that previously streamed cleanly.
- Any test that asserts nid values appear as attributes in mirror DOM.

**Phase to address:**
Node identity rework phase, sequenced *before* shadow DOM mirroring (shadow trees can't
be addressed by document-level selector queries at all, so the Map-based renderer lookup
is a prerequisite).

### Pitfall 9: Shadow DOM mirroring scoped as "walk one more tree" when it's a protocol change

**What goes wrong:**
Shadow DOM support gets estimated as a serializer tweak and turns into a multi-front
slog — rrweb's tracker shows years of incremental fixes (rrweb-snapshot #71 "Support for
shadow dom is incomplete", #933 custom-element misclassification, missed
`adoptedStyleSheets` of shadow roots in snapshots, duplicate rendering from slot
reparenting). Specific traps:

- **Closed roots are unreachable** without patching `Element.prototype.attachShadow` to
  retain a reference at creation time — which only captures roots created *after* the
  capture script loads; pre-existing closed roots are permanently invisible.
- **Slots**: light-DOM children render *inside* slots visually but live *outside* the
  shadow tree structurally. Serializing both the composed view and the shadow tree
  duplicates content; the mirror must reproduce the tree structure (host children +
  shadow root) and let slot projection happen naturally — which requires the mirror to
  *construct real shadow roots*, not flatten them into plain HTML.
- **MutationObserver doesn't cross shadow boundaries**: each shadow root needs its own
  observer, attached on discovery and on dynamic `attachShadow`.
- **Flattening breaks style scoping**: shadow CSS (`:host`, `::slotted`) only works
  inside real shadow roots; a flattened serialization renders wrong even if structure
  looks right.
- **Iframe recursion compounds**: same-origin iframes each have their own document,
  stylesheet set, and shadow trees; rrweb mirrors them with per-frame recording stitched
  by node ID, and cross-origin frames stay opaque (already out of scope here — keep it
  that way; rrweb's cross-origin recipe documents the postMessage trust problems that
  follow otherwise).

**Why it happens:**
The serializer's current single-TreeWalker design assumes one contiguous tree. Shadow DOM
makes the page a forest, with per-root observers, per-root adopted sheets, and a
composed-vs-structural distinction that doesn't exist in flat HTML.

**How to avoid:**
- Scope v1 deliberately: open shadow roots, serialized as structured ops (a `shadow` flag
  or dedicated op so the renderer attaches a real shadow root), per-root observers,
  per-root `adoptedStyleSheets`. Closed-root capture via `attachShadow` patch as a
  documented best-effort option; declarative shadow DOM via `getHTML({
  serializableShadowRoots: true })` where available.
- The `add` op's raw-HTML payload cannot express shadow boundaries — expect a protocol
  extension, and design it when doing the WeakMap addressing rework (Pitfall 8), not
  after.
- Fixtures: a real component-library page (e.g., anything Lit-based), a slot-heavy page,
  nested shadow roots.

**Warning signs:**
- Shadow DOM estimated at less than the truncation system's original effort.
- Plan says "flatten shadow trees into the snapshot HTML."
- No per-shadow-root observer in the design sketch.

**Phase to address:**
Shadow DOM phase, after node-identity rework; protocol extension designed jointly with
Pitfall 8.

### Pitfall 10: Extension-adapter state and watchdogs quietly die with the MV3 service worker

**What goes wrong:**
During extraction, the MV3-specific survival machinery gets abstracted into a generic
"host adapter" and loses the properties that made it work. MV3 realities: the service
worker is killed after ~30 s of inactivity (event arrival resets the timer); **all SW
global variables are lost on eviction**; `chrome.alarms` minimum period is 30 s from
Chrome 120 (1 minute before that — the reference's 1-minute alarm is safe either way);
content scripts outlive the SW but throw "Extension context invalidated" on
`chrome.runtime` calls after an extension reload/update, and bfcache can resurrect pages
whose message ports are dead. The reference's design is *already correct* (alarm-based
watchdog re-armed idempotently on every dispatch, session-stamp resync on eviction,
CONCERNS.md notes the `nextNodeId` reset is acceptable *because* the watchdog forces a
fresh snapshot) — the pitfall is losing these invariants in translation.

**Why it happens:**
The transport abstraction makes `chrome.runtime.sendMessage` look like any other `send()`.
But the extension adapter has lifecycle semantics no other adapter has: its "server side"
(the SW) evaporates mid-session. A generic adapter holding pending intents, session maps,
or diagnostics in SW module scope works in testing (SW stays alive under DevTools!) and
fails in production.

**How to avoid:**
- Write the extension adapter's invariants down before extracting: (1) watchdog state
  lives in `chrome.alarms`, never in SW globals; (2) anything the SW must remember across
  eviction goes to `chrome.storage.session`; (3) content-script side wraps every
  `chrome.runtime` call to catch context-invalidation and tear down observers cleanly;
  (4) eviction recovery = full resync via snapshot, by design — keep the
  `ext:request-snapshot` path.
- Test with the SW actually evicted: a Playwright/manual procedure that force-stops the
  worker (chrome://serviceworker-internals or the 30 s wait) mid-stream and asserts the
  mirror recovers. DevTools-open testing masks eviction entirely.
- Document for adapter authors that `Transport` implementations may be torn down and
  re-created mid-session; the capture core must tolerate transport replacement.

**Warning signs:**
- Adapter code with module-level `let pending…` in the SW context.
- All extension testing done with the SW DevTools inspector open.
- The chrome.alarms watchdog missing from the extracted adapter's first version.

**Phase to address:**
Host adapters phase (extension adapter); eviction-recovery test in the FSB swap-in
verification gate.

### Pitfall 11: npm packaging breakage — exports map, stale `.d.ts`, and silent file omission

**What goes wrong:**
Several independent, well-documented failure modes stack up for a plain-ESM,
JSDoc-typed package:

- **Exports map mistakes**: missing `"./package.json"` subpath export (breaks tooling
  that reads it); `types` condition not listed before `import`/`default` (TypeScript
  resolves the wrong file); subpaths consumers need (`./capture`, `./renderer`,
  `./protocol`) not exported, breaking deep imports the docs advertise.
- **Dual-package hazard**: if a CJS build is added later "for compatibility," CJS and ESM
  copies get separate module caches — two instances of module state (the protocol
  envelope's codec registry, any singleton) and identity-based checks fail. For a package
  whose constraint is *no build step*, shipping ESM-only avoids the hazard entirely;
  modern Node (20.17+/22.12+) can `require()` ESM anyway.
- **JSDoc→`.d.ts` drift**: `tsc` *emits declarations even when type-checking fails*
  (microsoft/TypeScript #36169 behavior) — broken `.d.ts` ships unless CI fails on
  errors first; and TypeScript treats a `.d.ts` sitting next to a same-name `.js` as
  authoritative input, so generated declarations emitted in-place go stale and then
  block regeneration. Emit to a separate `types/` directory and point `types`
  conditions there.
- **Files allowlist silent failures**: `files` entries that don't exist are skipped with
  no warning; `.gitignore` excluding `dist/`/`types/` leaks into packing when no
  `.npmignore` exists; `.npmignore`, if added, *replaces* `.gitignore` semantics
  entirely.
- **Provenance**: `--provenance` only works from supported cloud CI (GitHub Actions /
  GitLab); a local `npm publish` from a laptop can't attach it. Trusted publishing
  (OIDC) generates provenance automatically and removes long-lived tokens.

**Why it happens:**
Packaging correctness is invisible locally — `npm link` and in-repo tests resolve through
the filesystem, not the exports map or the packed tarball.

**How to avoid:**
- CI gate before every publish: `npm pack --dry-run` (inspect file list), `publint`, and
  `arethetypeswrong` (`attw --pack`) against the actual tarball; a smoke test that
  installs the tarball into a scratch project and imports every advertised subpath in
  Node and a bundler.
- Decide **ESM-only** explicitly and early; document the Node floor that supports
  `require(esm)` for CJS consumers.
- Declaration build: `tsc --checkJs --declaration --emitDeclarationOnly --outDir types`,
  with the type-check step failing CI on any error *before* emit is used.
- Publish only from GitHub Actions with trusted publishing/provenance from the first
  release — retrofitting provenance after consumers exist invites version-trust questions.

**Warning signs:**
- "Works with `npm link` but not when installed from the registry."
- A `.d.ts` file sitting beside its `.js` source in `src/`.
- First publish performed from a developer laptop.

**Phase to address:**
npm packaging/publication phase; the tarball smoke test becomes a permanent CI job.

### Pitfall 12: Premature API freeze — publishing the SDK surface before the FSB swap-in proves it

**What goes wrong:**
The package is published (and announced) with an API designed during extraction; then the
FSB swap-in — the only real consumer — reveals the `Transport` interface is missing a
capability (readiness probing, pause/resume semantics, the pending-intent re-arm), and
the fix is a breaking change against a published surface. The inverse failure also
happens: the API is shaped *so* tightly around FSB's needs (overlay provider, dialog
cards, skipElement) that the "plug-and-play framework" story is FSB-shaped with the
serial numbers filed off, and the two demos feel bolted on.

**Why it happens:**
Extraction order pressure: npm publication is a visible milestone and tempting to ship
before the consumers (FSB swap-in, two-tab demo, Playwright demo) have exercised the API.
With one consumer, every coupling looks like a requirement.

**How to avoid:**
- Sequence three consumers before 1.0: two-tab demo and Playwright demo (generic
  consumers) *plus* FSB swap-in (demanding consumer). Publish `0.x` and say loudly that
  0.x means churn; reserve 1.0 until all three run on the same published API.
- Make FSB-specific needs (overlays, dialogs, skip predicates) *extension points*
  (options/hooks) rather than core API, and prove it by the generic demos not using them.
- Keep the wire protocol versioned independently of the JS API — FSB compatibility lives
  in the protocol (`{_lz, d}`, session stamps), and CONCERNS.md's fragile
  "missing-identity = accepted" compatibility bypass must be hardened in the same change
  that bumps the protocol version, not separately.

**Warning signs:**
- npm publish scheduled before either demo exists.
- `Transport` interface has FSB-named concepts in it.
- A breaking JS-API change is forced by the swap-in after 1.0.

**Phase to address:**
Publication phase ordering: demos → 0.x publish → FSB swap-in → 1.0.

### Pitfall 13: Evaluation that can't be reproduced or compared fairly — the reviewer-killer set

**What goes wrong:**
Four distinct mistakes, each independently fatal in review:

1. **Live-web corpus**: pages change between runs (ads, A/B tests, CDN rotation), so
   bandwidth/fidelity numbers aren't reproducible even by the authors. The systems-field
   answer is record-and-replay (Mahimahi, SIGCOMM 2014, built exactly for this:
   preserves multi-origin structure, isolates traffic; WebPageReplay is the
   Chromium-native alternative) with a *frozen, versioned* corpus and scripted,
   deterministic activity levels.
2. **Baseline unfairness**: WebRTC adapts bitrate/resolution under congestion control —
   reporting its bandwidth without its delivered visual quality (or vice versa) is
   comparing different operating points. CDP screencast has quality/format/`everyNthFrame`
   parameters and a frame-ack flow-control loop; defaults are not a neutral choice and
   must be reported. rrweb has an optional `packFn` compression step. Comparing
   PhantomStream's LZ-string-compressed wire bytes against rrweb's *uncompressed* JSON
   events (or against WebSocket permessage-deflate'd anything) is a
   compressed-vs-uncompressed channel mismatch — report both raw payload bytes and
   on-the-wire bytes for every system, with identical compression policy or explicit
   per-system disclosure.
3. **SSIM misuse**: SSIM was designed for natural images; the research literature
   (including "Understanding SSIM," arXiv 2006.13846) documents nonintuitive and
   misleading scores — notably on text-heavy/synthetic content and under small spatial
   misalignment, which is *exactly* what a DOM mirror vs. source-tab comparison produces
   (font rasterization differences, scrollbar offsets, 1-px layout shifts). A
   pixel-metric-only fidelity story will be attacked. Complement with DOM/semantic-level
   fidelity (element-presence/attribute/text diff rates, interactable-element
   addressability — which is the system's actual value proposition) and define the
   pixel-metric protocol precisely (viewport normalization, font availability, capture
   timing).
4. **Missing statistical rigor**: single-run numbers, no variance, no run count, no
   environment spec. Systems-track norms: n repeated runs, report median + dispersion,
   fixed hardware/browser versions, harness and corpus released as artifacts.

**Why it happens:**
The evaluation harness doubles as the framework's regression suite, so it grows
incrementally from dev convenience scripts — which are exactly the wrong defaults for
reproducibility (live sites, one run, default baseline configs).

**How to avoid:**
- Stand the harness up on a frozen replay corpus from day one of the evaluation phase;
  treat corpus version + browser version + harness commit as the experiment's identity
  triple.
- Write a one-page **baseline-configuration protocol** before collecting numbers:
  for each baseline, which knobs, set to what, why, and what "equal footing" means
  (same viewport, same activity script, same network shape via LinkShell/DelayShell or
  equivalent).
- Choose metrics so the headline claims survive: bandwidth (raw + wire), latency
  (event-to-mirror, defined precisely), pixel fidelity (with protocol), semantic fidelity
  (DOM-diff based — none of the pixel baselines can even score on it, which is the
  paper's argument made measurable).
- Decide run counts and statistics in the harness design, not at writing time.

**Warning signs:**
- Harness scripts contain live URLs.
- Baseline invocations rely on undocumented defaults.
- Results tables with single numbers and no ± anywhere.
- SSIM is the only fidelity metric in the draft.

**Phase to address:**
Evaluation harness phase (corpus + protocol first, numbers second); paper phase inherits
the artifact-release requirement.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Skip differential harness, eyeball mirror output | Faster first extraction | Undetectable behavior drift; FSB swap-in becomes the (late, expensive) test | Never — the reference is the project's main asset |
| Keep `data-fsb-nid` renderer lookups while adding WeakMap capture-side | Smaller diffs per PR | Halfway identity state where ops miss silently | Only inside one PR series, never across a release |
| Sanitize at capture only, defer render-side | One sanitizer integration | mXSS exposure; capture side is attacker-context anyway | Never for published versions |
| Snapshot-only CSSOM mode (no style ops) | Big scope cut | Recreates frozen-style drift one level up | OK for v1 if loudly documented + in paper limitations |
| Publish before demos/swap-in exercise the API | Visible milestone | Breaking changes against a public surface | OK only under 0.x with explicit instability notice |
| Add CJS build "for compatibility" | Wider consumer support | Dual-package hazard, doubled test matrix | Never — ESM-only + `require(esm)` floor instead |
| Evaluation on live sites during development | Zero corpus setup | Numbers are irreproducible; must redo all runs later | OK for smoke tests only, never for reported numbers |
| Keep the "missing-identity = accept" filter bypass as-is | FSB back-compat preserved | Unauthenticated message acceptance path in a published framework | Only until the protocol-version bump; harden in that change |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Chrome MV3 SW | State in SW globals; alarms < 30 s; testing with DevTools open (suppresses eviction) | `chrome.storage.session` for state; alarm ≥ 30 s (Chrome 120+ floor, 1 min before); eviction-forcing test procedure |
| Content script ↔ SW | Unguarded `chrome.runtime` calls after extension reload ("Extension context invalidated"); bfcache-resurrected pages with dead ports | Wrap runtime calls, catch invalidation, tear down observers, rely on watchdog resync |
| CDP adapter (Playwright) | Assuming content-script semantics; `addInitScript` runs before DOM exists; frame navigations silently replace the injected world | Re-inject on navigation events; session lifecycle tied to frame lifecycle; use CDP `CSS.getStyleSheetText` as the cross-origin CSS fallback |
| DOMPurify | String-in/string-out then `innerHTML` (mXSS roundtrip); loosening `ALLOWED_URI_REGEXP`/hooks | `RETURN_DOM_FRAGMENT`, insert nodes directly; stock config + explicit allowlist additions only |
| Sanitizer API | Treating it as available baseline | Limited availability (Chrome 146/Firefox 148); feature-detect, DOMPurify fallback |
| lz-string | Re-vendoring the unversioned minified blob | npm dependency via injected codec (envelope already supports this); test real codec, not just `fakeLz` |
| WebSocket relay | Relaying parsed JSON without schema validation (CONCERNS.md) | Type allowlist + payload shape check in `limits.js` before fan-out |
| npm registry | First publish from laptop; no tarball verification | Trusted publishing from CI with provenance; `npm pack --dry-run` + `attw` + tarball install smoke test |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Per-op `querySelector('[data-fsb-nid=…]')` in renderer | Mirror lags on mutation bursts | `Map<nid, Element>` maintained on add/rm (falls out of Pitfall 8) | Thousands of nodes × high-frequency batches (already flagged in CONCERNS.md) |
| `clone.innerHTML.length` re-evaluated per truncation iteration | Snapshot stalls on big pages | Running byte estimate or one re-measure per pass | Large pages needing many subtree drops |
| Full-enumeration computed styles (regression of the curated list) | Multi-second serialize on heavy pages | Keep CURATED_PROPS + elision; budget test in CI | The original 45 s YouTube case |
| Per-element `getComputedStyle` for post-snapshot added nodes (new feature) | Add-op bursts cause jank | Batch style reads per rAF flush; reuse single pass | Infinite-scroll/feed pages adding hundreds of nodes |
| Per-shadow-root observers leak on detached roots | Memory growth on SPA navigation | Track roots in WeakSet; disconnect on host removal | Long-lived SPA sessions |
| CSSOM mode re-serializing whole sheets per style op | Style-op payloads dwarf DOM diffs | Rule-level ops addressed by sheet ID + index | CSS-in-JS pages inserting rules continuously |
| `roomDiagnostics` unbounded growth in relay | Relay RSS climbs forever | Prune with room removal (CONCERNS.md) | Long-running relay, many unique sessions |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| No `sandbox` on the mirror iframe (current reference state) | Captured-page script executes in viewer origin | `sandbox="allow-same-origin"` (+`allow-forms` at most), asserted at startup |
| Adding `allow-scripts` alongside `allow-same-origin` | Framed content removes its own sandbox; full consumer-origin XSS | Hard invariant; `UNSAFE_`-prefixed opt-in if ever needed |
| Capture-side-only sanitization | mXSS via serialize→parse roundtrip; attacker controls capture context | Sanitize both ends; render-side `RETURN_DOM_FRAGMENT`; sandbox as backstop |
| Raw `<style>`/rule text into srcdoc | CSS exfiltration beacons, `@import` of attacker sheets | CSS sanitizer pass + CSP meta in srcdoc (`script-src 'none'`, scoped `style-src`/`img-src`/`font-src`) |
| String-concatenated shell attributes (`buildShellAttributeString`, only `"` escaped) | Malformed/escaping-confused srcdoc → injection | Build shell via `createElement` + `setAttribute` |
| "Missing identity = accept" filter bypass kept in published framework | Unauthenticated message injection into viewer | Harden with protocol version bump; back-compat only behind explicit FSB-compat flag |
| Relay relays unvalidated payloads | Malicious client poisons all viewers in room | Schema/type allowlist in `limits.js`; size caps already exist |
| Cross-origin iframe capture "workaround" added later | Any embedding page can record your content; postMessage events interceptable (rrweb's documented warning) | Keep cross-origin frames out of scope; document why, citing rrweb's recipe |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Mirror silently stale (truncation misses, frozen styles) | Viewer trusts an out-of-date view — dangerous for remote control | Surface staleness: `staleMutationCount`/resync events exposed in renderer API; visible "resyncing" state |
| Remote control enabled while mirror is stale or mid-resync | Clicks land on wrong elements in the real tab | Gate reverse-path input on sync state; coordinate-mapping only against current snapshot ID |
| Hardcoded resync threshold (3 misses) thrashing on fast-mutating pages | Constant flicker/re-snapshot on tickers/terminals | Configurable threshold + debounce window (CONCERNS.md fragile area) |
| Embeddable renderer assumes FSB dashboard CSS/layout | Broken layout in consumer apps | Self-contained styles; layout modes as documented options; demo apps as the proof |
| `npx phantom-stream demo` requiring manual relay setup | First-touch experience fails | Bundled relay, zero-config demo as an explicit acceptance test |

## "Looks Done But Isn't" Checklist

- [ ] **Sanitization:** strips `on*`/`javascript:` in the snapshot path — verify the *same
  chokepoint* covers `processAddedNode`, attr-op re-absolutification, and the future
  subtree-fetch path; verify render-side sanitizes again (mXSS fixtures in tests).
- [ ] **Sandbox:** iframe has `sandbox` set — verify no `allow-scripts`, verify the
  assertion fires if a consumer passes an unsandboxed iframe, verify CSP meta present in
  srcdoc.
- [ ] **CSSOM mode:** works on fixture pages — verify against a *production-built*
  CSS-in-JS page (empty `<style>` text), a cross-origin-CDN page (SecurityError path),
  and a constructable-stylesheets page; verify `@import` chains flattened.
- [ ] **WeakMap identity:** capture side migrated — verify zero `data-fsb-nid` strings
  remain in renderer; verify text-node addressing decision documented; verify move
  semantics (same node removed+re-added keeps identity or resyncs cleanly).
- [ ] **Shadow DOM:** snapshots render — verify slotted content isn't duplicated, shadow
  CSS (`:host`, `::slotted`) applies (i.e., real shadow roots constructed in mirror),
  per-root observers attached on dynamic `attachShadow`.
- [ ] **Extension adapter:** streams in dev — verify with SW force-evicted mid-stream
  (DevTools closed), extension reloaded mid-stream (context invalidation), and alarm
  watchdog confirmed re-arming.
- [ ] **npm package:** `npm link` works — verify packed tarball installs and imports in a
  scratch project (Node + bundler), `attw`/`publint` clean, `types/` in files allowlist,
  provenance attached on the registry page.
- [ ] **Evaluation:** numbers collected — verify corpus is frozen/versioned, baseline
  knobs documented, raw vs wire bytes both reported, run count ≥ planned n with
  dispersion, pixel-metric protocol written down.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Hardening lost in extraction (1) | MEDIUM | Diff extracted module against `reference/` per the §5 checklist; re-port defenses; add the missing regression tests before further refactors |
| Behavior drift discovered late (2) | HIGH | Stand up differential harness retroactively; bisect divergences into ledger (intentional) vs bugfix backlog; freeze API until triaged |
| mXSS/sandbox vulnerability shipped (3, 4) | HIGH | Security release + advisory; deprecate vulnerable versions on npm; add the missing layer (sandbox assertion/render-side sanitizer) + fixtures; this is why both land pre-publication |
| CSSOM mode fails on real-world CSS (6, 7) | MEDIUM | Fallback chain degrades to current curated-inline mode per sheet; ship as mode flag so consumers pin behavior; fix forward |
| Identity desync after WeakMap rework (8) | MEDIUM | Session-stamp resync already recovers (full snapshot); fix minting/move rules; differential harness gates re-release |
| Published API breaks at FSB swap-in (12) | MEDIUM | 0.x breaking release with migration note; if post-1.0, additive shim + major bump |
| Evaluation invalidated (live corpus / unfair baseline) (13) | HIGH | Re-run everything on frozen corpus with documented configs — budget for one full redo; avoid by freezing corpus before any reported run |
