<objective>
Compress WebSocket payloads in the extension before sending through the relay, and decompress on the dashboard side before processing. This fixes the core problem: the relay server (fly.io) drops or fails to forward large WS messages, causing DOM stream snapshots (100KB+) and task results to never reach the dashboard.

Purpose: The relay is a blind passthrough with an apparent message size limit. Compressing payloads client-side (extension) and decompressing client-side (dashboard) avoids this limit without any relay changes.

Output: Modified ws-client.js send() that compresses large payloads, modified dashboard.js that transparently decompresses, vendored lz-string library.
</objective>

3. In ws/ws-client.js, modify the send() method (lines 123-127) to compress payloads above a size threshold:

   ```javascript
   send(type, payload) {
     if (!this.ws || this.ws.readyState !== WebSocket.OPEN) return false;
     var raw = JSON.stringify({ type, payload, ts: Date.now() });
     // Compress payloads larger than 1KB to avoid relay message size limits
     if (raw.length > 1024 && typeof LZString !== 'undefined') {
       var compressed = LZString.compressToBase64(raw);
       // Only use compression if it actually reduces size
       if (compressed.length < raw.length) {
         this.ws.send(JSON.stringify({ _lz: true, d: compressed }));
         return true;
       }
     }
     this.ws.send(raw);
     return true;
   }
   ```

   Key design decisions:
   - Threshold of 1024 bytes: pings (tiny), task-progress (small) skip compression. dom-snapshots (100KB+) and task-complete (varies) get compressed.
   - Envelope: { _lz: true, d: "base64data" } -- the _lz flag tells the dashboard this needs decompression. The relay sees a small JSON object, not the huge payload.
   - Fallback: if LZString is not loaded for some reason, send uncompressed (backward compat).
   - Size check: if compression makes it bigger (unlikely but possible for tiny payloads near threshold), send uncompressed.

4. Add a log line after compression for debugging:

   ```javascript
   console.log('[FSB WS] Compressed ' + type + ': ' + raw.length + ' -> ' + compressed.length + ' bytes (' + Math.round(compressed.length / raw.length * 100) + '%)');
   ```

   With:
   ```javascript
   ws.onmessage = function (event) {
     try {
       var envelope = JSON.parse(event.data);
       var msg;
       // Detect compressed payload from extension
       if (envelope._lz && envelope.d && typeof LZString !== 'undefined') {
         var decompressed = LZString.decompressFromBase64(envelope.d);
         if (!decompressed) {
           console.warn('[FSB-DASH] Failed to decompress WS message');
           return;
         }
         msg = JSON.parse(decompressed);
         console.log('[FSB-DASH] WS msg (decompressed):', msg.type, msg.payload ? JSON.stringify(msg.payload).substring(0, 100) : '');
       } else {
         // Uncompressed message (backward compatibility or small payloads)
         msg = envelope;
         console.log('[FSB-DASH] WS msg:', msg.type, msg.payload ? JSON.stringify(msg.payload).substring(0, 100) : '');
       }
       handleWSMessage(msg);
     } catch (e) {
       console.warn('[FSB-DASH] WS parse error:', e.message);
     }
   };
   ```

   Key points:
   - Check for _lz flag in the parsed envelope
   - LZString.decompressFromBase64 returns the original JSON string
   - Parse the decompressed string to get the real { type, payload, ts } message
   - If decompression fails (null return), log a warning and skip (don't crash)
   - If no _lz flag, treat as normal uncompressed message (backward compat)
   - Keep the same console.log format so existing debugging workflows work
