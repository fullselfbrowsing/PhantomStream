# 122.4 Debug Report: Dashboard Not Receiving Messages

## Summary

Two categories of messages fail to reach the dashboard:
1. **Compressed messages** (DOM snapshots, any payload >1KB) -- ROOT CAUSE CONFIRMED
2. **Task completion messages** -- Likely a consequence of issue #1, not a separate bug

---

## Root Cause #1: Browser Caching Stale dashboard.js (CONFIRMED)

### Evidence

**E1: Static file cache headers set to 1 day**

`server/server.js:75-78`:
```javascript
app.use(express.static(staticPath, {
  maxAge: '1d',
  etag: true
}));
```

This tells browsers to cache ALL static files (JS, CSS, images) for 24 hours without revalidation. Once a browser downloads `dashboard.js`, it will NOT request it again for 1 day, even after a fly.io redeploy.

**E2: No cache-busting on JS files**

`showcase/dashboard.html:482-484`:
```html
<script src="js/lz-string.min.js"></script>
<script src="js/main.js"></script>
<script src="js/dashboard.js"></script>
```

Compare with the CSS which HAS a version parameter:
```html
<link rel="stylesheet" href="css/dashboard.css?v=20260329">
```

The JS files have NO `?v=` query parameter. Browsers with cached copies will never request the new version until the 24-hour TTL expires.

**E3: Console log proves old code is running**

The user sees:
```
[FSB-DASH] WS msg: undefined ""    (x3)  -- at line 2116
```

- In the **current** dashboard.js, the uncompressed log is at line 2391, and the decompression code is at line 2380.
- Line 2116 in the current file is an empty line inside a ResizeObserver block.
- This proves the browser is executing an **older version** of dashboard.js where `console.log('[FSB-DASH] WS msg:', msg.type, ...)` was at line 2116.

**E4: The old code can't handle compressed envelopes**

The old `ws.onmessage` (before commit `53ac78b`) was:
```javascript
var msg = JSON.parse(event.data);
console.log('[FSB-DASH] WS msg:', msg.type, ...);
handleWSMessage(msg);
```

When it receives `{ _lz: true, d: "..." }`:
- `msg.type` = `undefined` (no `type` property on the `_lz` envelope)
- `msg.payload` = `undefined`
- `handleWSMessage(msg)` matches no handler and silently drops the message

**E5: The relay server does NOT modify messages**

`server/src/ws/handler.js:34-43` parses JSON only to check for `type === 'ping'`. For compressed messages where `type` is undefined, it correctly falls through to `relayToRoom(hashKey, ws, data.toString())` which passes the raw string unchanged.

**E6: No WS message size limits in play**

- The `ws` npm library (v8.19+) defaults to 100 MiB maxPayload -- far larger than any DOM snapshot.
- The relay `WebSocketServer` is created with `{ noServer: true }` and no custom `maxPayload`.
- Fly.io's WS proxy passes through TCP frames without size limits (HTTP body limits only apply to REST, not upgraded WS connections).

### Mechanism

```
Extension sends compressed message:
  ws-client.js send() -> JSON.stringify({ _lz: true, d: "..." })
    -> relay server receives -> parses OK -> relays raw string unchanged
      -> dashboard browser receives -> OLD dashboard.js parses JSON
        -> msg.type === undefined -> logs "undefined" -> handleWSMessage drops it
```

## Root Cause #2: Task Completion Not Reaching Dashboard

### Analysis

The `ext:task-complete` payload is small (~120 bytes), well under the 1KB compression threshold. It is sent UNCOMPRESSED. The old cached dashboard.js CAN parse uncompressed messages correctly.

**Three possible explanations:**

**A. The user may be describing a consequence of the DOM stream failure, not a separate message loss.** If the DOM stream preview is stuck on "Connecting to browser...", the overall dashboard experience feels broken. The task progress DOES arrive (user confirmed `ext:task-progress` works), so `ext:task-complete` should also arrive since it uses the same uncompressed path.

**B. WS connection may drop during long tasks.** Fly.io may close idle connections. The dashboard has NO keepalive ping (the extension does, but only on its side). If Fly.io's proxy considers a connection idle when no client-to-server frames are seen from the dashboard side, the connection could be terminated mid-task. Evidence: the dashboard code has no `ping` or keepalive interval -- only the extension sends pings.

**C. The `broadcastDashboardComplete` dedup flag.** If the user previously sent a `dash:stop-task`, the `_dashStopSent` flag is set to `true` (ws-client.js:390). If the task completes naturally in the same instant, `broadcastDashboardComplete` (background.js:940) checks this flag and SKIPS sending the completion. This is a race condition that could cause silent message loss.

### Verdict

Most likely explanation A. The user sees task-progress arriving (small payloads work), so the WS channel is alive. If `ext:task-complete` truly doesn't arrive, explanation B (no dashboard keepalive) is the next suspect.

## Root Cause #3 (Minor): No Dashboard-Side Keepalive

### Evidence

- `ws/ws-client.js:146-151` -- Extension sends ping every 20 seconds
- `showcase/js/dashboard.js` -- NO keepalive/ping anywhere in the file
- Fly.io idle timeout could close the connection if the dashboard doesn't send any frames for an extended period

### Impact

During a long-running task where the dashboard is just receiving `ext:task-progress`, the dashboard never sends frames back. Fly.io or intermediate proxies may close the connection. This would cause both DOM stream and task completion messages to stop arriving.

## Recommended Fixes (Priority Order)

### Fix 1: Cache-busting for JS files (CRITICAL)

**File: `showcase/dashboard.html`**

Add version query parameters to all JS files:
```html
<script src="js/lz-string.min.js?v=20260329"></script>
<script src="js/main.js?v=20260329"></script>
<script src="js/dashboard.js?v=20260329"></script>
```

### Fix 2: Reduce static file max-age (CRITICAL)

**File: `server/server.js`**

Change cache duration for JS files to be much shorter, or use `no-cache` with etag revalidation:

Option A -- Short cache with revalidation:
```javascript
app.use(express.static(staticPath, {
  maxAge: 0,
  etag: true
}));
```

Option B -- Different policy for JS vs other static assets:
```javascript
app.use('/js', express.static(path.join(staticPath, 'js'), {
  maxAge: 0,
  etag: true
}));
app.use(express.static(staticPath, {
  maxAge: '1d',
  etag: true
}));
```

### Fix 3: Dashboard keepalive ping (RECOMMENDED)

**File: `showcase/js/dashboard.js`**

Add a keepalive inside `connectWS()`:
```javascript
// Inside ws.onopen:
wsKeepaliveTimer = setInterval(function() {
  if (ws && ws.readyState === WebSocket.OPEN) {
    ws.send(JSON.stringify({ type: 'ping', ts: Date.now() }));
  }
}, 20000);

// Inside ws.onclose and disconnectWS:
if (wsKeepaliveTimer) { clearInterval(wsKeepaliveTimer); wsKeepaliveTimer = null; }
```

### Fix 4: Force existing users to get fresh assets (IMMEDIATE)

After deploying fixes 1 and 2, existing users with cached stale JS will still be stuck until their cache expires. Options:
- Add `Cache-Control: no-cache` header specifically for `dashboard.js` and `lz-string.min.js` as a temporary override
- Or rename the files (e.g., `dashboard.v2.js`) to bypass all caches
