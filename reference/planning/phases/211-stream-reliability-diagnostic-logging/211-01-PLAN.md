## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-211-01-01 | Tampering | inbound onmessage `_lz` branch | mitigate | Decompression failure (null/empty) is detected explicitly via `if (!decoded) return after recordFSBTransportFailure(...)`. The frame is dropped, not partially dispatched. JSON.parse of the decoded string is wrapped in the existing outer try/catch which logs and drops. No partial state is mutated before parse success. |
| T-211-01-02 | Denial of Service | LZString.decompressFromBase64 -- decompression bomb (output orders of magnitude larger than input) | mitigate | LZString output is bounded by the algorithm's lookback window and the `d` string length; a 1KB compressed input cannot expand to gigabytes the way deflate-bomb attacks can (LZString lacks the back-reference depth deflate exploits). The relay also imposes a per-message size limit at `server/src/ws/handler.js`. The frame is single-shot per message — no state accumulates across frames (D-16: stateless per-frame). Residual: an attacker producing many large `_lz` frames can drive CPU; the existing message-loop is single-threaded and naturally rate-limits. Out-of-scope to add an explicit byte cap in this plan; revisit if telemetry shows abuse. |
| T-211-01-03 | Tampering | JSON.parse(decoded) | mitigate | JSON.parse rejects non-JSON syntax. The outer try/catch at the existing `console.warn('[FSB WS] Failed to parse message:', ...)` line catches and drops. The decoded inner object is then dispatched via `_handleMessage(raw)` which routes on `raw.type` — unknown types fall through the existing switch as a no-op. No `eval`, no `new Function`, no prototype lookup on raw. |
| T-211-01-04 | Information Disclosure | recordFSBTransportFailure payload | mitigate | The failure payload includes only: `target: 'inbound'`, `type: '_lz'`, `tabId: number`, `error: <fixed-string>`, `len: raw.d.length`. NO sample of the malformed `d` content. NO stack trace. The transport-failure ring buffer is local-only (does not leave the SW until 211-03's exportDiagnostics handler is added), and even then only the structured shape is exported. |
| T-211-01-05 | Spoofing | Inbound `_lz` envelope from non-relay source | accept | Any peer with a valid WebSocket connection can send any frame. The relay's auth model (server/src/ws/handler.js) is the connection-level boundary; this plan does not alter it. Compressed frames receive identical trust to plain JSON frames, which is the correct symmetric behavior. |
| T-211-01-06 | Repudiation | Decompression failures are recorded but with limited fidelity | accept | recordFSBTransportFailure stores `{event, ts, type, target, tabId, readyState, error, len}` in an in-memory ring (ws-client.js:86-96). Sufficient for post-hoc triage in `chrome.runtime` debug surfaces. No persistent audit log is required for v0.9.45rc1 transport telemetry. |
| T-211-01-07 | Elevation of Privilege | LZString.decompressFromBase64 prototype pollution via crafted base64 | mitigate | LZString is a pure-function library that operates on string -> string transforms; it does not access `Object.prototype` or perform key lookups on user input. The decoded output is a string, then JSON.parse returns a fresh plain object. Prototype pollution would need to land in `_handleMessage`'s downstream code, which is unchanged by this plan and already validated by `tests/remote-control-handlers.test.js`. |
