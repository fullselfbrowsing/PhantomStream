# Phase 122.2: Stop Signal & Final Outcome - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Dashboard stop button reliably halts any running FSB automation (regardless of origin), and task completion/failure results are relayed back to the dashboard with correct final state display. Fixes the broken stop path where `_handleStopTask` only flags `session.status` without actually stopping the automation loop.

</domain>

<decisions>
## Implementation Decisions

### Stop mechanism
- **D-01:** Hard stop -- wire `dash:stop-task` to call `handleStopAutomation()` in background.js, which kills the iteration loop, cancels pending AI calls, and cleans up content script observers. Instant halt, no graceful wait.
- **D-02:** The current `_handleStopTask()` in ws-client.js must be rewritten to call `handleStopAutomation()` (or send `{ action: 'stopAutomation' }` to the active tab) instead of just setting `session.status = 'stopped'`.

### Task scope
- **D-03:** Stop button halts ANY running automation, not just dashboard-submitted tasks. Remove the `_isDashboardTask` filter in `_handleStopTask`. If the user sees something running and clicks Stop, it should stop.

### Result display
- **D-04:** On success: show AI result summary text + elapsed time. This is already implemented in `handleTaskComplete` / `setTaskState('success')` -- the fix is ensuring `broadcastDashboardComplete()` fires reliably on ALL completion paths (normal, error, timeout, max iterations).
- **D-05:** No additional metrics (tokens, cost, iterations) in this phase -- keep it to summary + time.

### Stopped task outcome
- **D-06:** When user clicks Stop mid-task, show "Stopped by user" plus partial progress -- the last action summary from `ext:task-progress`. The dashboard already receives progress updates, so the most recent one should be preserved and displayed alongside the stopped state.

### Claude's Discretion
- Whether to add a brief "Stopping..." intermediate state in the UI before the stop completes
- Exact cleanup sequence in the stop path (cancel AI call first, then content script, or parallel)
- Whether to deduplicate the `ext:task-complete` message if both `handleStopAutomation` and `_completionCallback` try to send one

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `handleStopAutomation()` in background.js: Proper stop implementation already exists -- just needs to be called from the dashboard stop path
- `broadcastDashboardComplete()`: Already sends `ext:task-complete` with proper payload -- needs to cover all completion paths
- `setTaskState('success'|'failed')`: Dashboard state machine already handles both outcomes with UI rendering
- `handleTaskComplete()`: Already parses `ext:task-complete` payload and routes correctly

### Established Patterns
- WS messages: `{ type: 'ext:*' | 'dash:*', payload: {...}, ts: Date.now() }`
- Stop in MCP path: `mcp:stop-automation` -> `{ action: 'stopAutomation' }` to content script -> session cleanup. Dashboard path should mirror this.
- Task lifecycle: `dash:task-submit` -> `ext:task-progress` (repeated) -> `ext:task-complete`

### Integration Points
- `ws/ws-client.js` `_handleStopTask()`: Rewrite to call handleStopAutomation or send stopAutomation message to active tab
- `background.js`: Ensure broadcastDashboardComplete fires on all exit paths (normal completion, error, timeout, max iterations, stop)
- `showcase/js/dashboard.js`: Preserve last `ext:task-progress` action text for display when task is stopped

</code_context>

<specifics>
## Specific Ideas

- The stop should feel instant -- no visible delay between clicking Stop and seeing the UI update
- When stopped, showing the last progress action gives context about what was happening ("Was clicking submit button..." vs just "Stopped")
- The stop button should work for MCP tasks too -- if you can see it running in the dashboard, you should be able to stop it

</specifics>

<deferred>
## Deferred Ideas

- Detailed metrics display (tokens, cost, iterations) on task completion -- could be a future enhancement
- Task pause/resume functionality -- different from stop, would need state preservation
- Stop confirmation dialog -- decided against for instant feel, but could revisit if accidental stops become an issue

</deferred>
